# 前端设计说明

> 这份是给"接手维护"的人看的——记录**这些页面为什么长这样，避免你想"改回去"的时候踩之前已经踩过的坑**。

---

## 1. 五张页面 & 它们的角色

| 页面 | 角色 | 关键交互 |
|---|---|---|
| **概览 Overview** | 每天瞄一眼的运营仪表 | KPI 4 卡 + 调用量折线 + 工具排行 Top 10 + Agent 占比 + 最近异常 + 系统健康 |
| **技能 Skills** | 控制台最重要的页面，会被用得最多 | 用户自定义场景分组（拖拽排序）+ 启停开关 + 写入许可 + **「对哪些 agent 可见」下拉** |
| **Agents** | 登记可接入的 AI 客户端 | 列表 + 新建（一次性 secret 弹窗）+ 启停 + 轮换密钥 |
| **远程接入 Remote** | 把第三方 MCP 服务器并入中枢 | 服务器卡片 + 添加表单 + 重新发现 |
| **审计 Audit** | 起居注，有事的时候才看 | **按会话折叠**（不是 raw log）+ 按 agent/异常过滤 + 搜索 |

## 2. 关键设计决策（"为什么这样"）

### 2.1 技能可见性放在「技能页」，不是「Agents 页」

之前的旧控制台把"按 agent 配置工具可见性"放在 Agents 页——
- 进 Agents 页 → 点某个 agent → 弹个对话框 → 在里面勾工具

我们改成：技能页**每一行**右边一个"对哪些 agent 可见"下拉。
- 默认所有 agent 都打勾 = 全部可见，按钮显示 `● 全部可见 ▾`，安静的灰底
- 取消任一个勾 → 按钮变成 `● 3 / 6 可见 ▾`，朱粉色高亮，一眼能看出"这行被限制过"
- 没有限制时**几乎隐形**，有限制时**自动凸显**

为什么这样：用户实际操作时不是「这个 agent 该有哪些工具」，而是「这个工具应该让谁看到」——后者更频繁、更自然。

### 2.2 技能分类用「用户自定义场景分组」，不是后端 category

后端的 `category` 字段是按**功能维度**分（realtime / action / finance / personal / search / remote / meta）。
但你说的对：从 MCP 本质讲，**所有工具都是动作**，「action」这个分类不准确。

所以前端：
- 后端的 `category` 退化为每行的小标签 chip（不抢戏）
- **页面章节用用户自己定义的场景分组**：「日常感知」「写作记录」「资产查询」「调试常用」……
- 用户可改名、可拖拽排序、可删除
- 一个技能只能在一个组里
- 不在任何组里的 → 自动落到页面底部的「未归类」分区

这套分组数据**需要后端给个端点存**（详见 BACKEND_TODOS.md §P1），前端有 localStorage 兜底。

### 2.3 审计页改造为「会话故事」

旧版审计页是 raw log：
```
2026-06-06T08:46:19.169Z mcp_request claude-ai ✓
2026-06-06T08:46:19.145Z mcp_request claude-ai ✓
2026-06-06T08:46:09.081Z mcp_request claude-ai ✓
...
```
人读起来累，**给机器调试用还行，给人看是失败的**。

我们的改造：
- **同一 agent / 60 秒内 / 同一成功状态**的连续日志折叠成一个 **session card**
- 每张卡片上面是一句"人话"：「Claude.ai 连接，进行了 5 次操作」
- 下面是工具调用 chip 横排
- 失败的 session 用红底高亮
- 按日分组，今天 / 昨天 / 6·4 这种相对日期

折叠逻辑全在前端做（见 `src/pages/Audit.tsx > groupIntoSessions`），不需要后端做任何配合。

### 2.4 概览页的 KPI 是"运营指标"，不是"工程指标"

设计概览页时调研了 Cloudflare AI Gateway / LiteLLM / Composio / Smithery 的仪表。它们都共享一个模式：

**给非工程师看的 4 个 KPI**：
- 周期调用数（带与上周期对比）
- P95 延迟
- 错误 / 拦截数
- 活跃 Agent 数

外加四张卡：调用量趋势、工具排行 Top 10、Agent 占比、最近异常。

这一套全靠 `GET /api/console/stats?range=…`（**待后端实现**，BACKEND_TODOS.md §P3）。
后端如果嫌麻烦不做也行——前端会优雅显示「等待后端实现」占位，不会崩。

## 3. 风格层

### 3.1 樱羽 Sakura（asashiki-design v0.1）

直接复用 `asashiki-design/tokens/tokens.css`。配色：
- 主底白 `#fff`
- 印象底色：浅粉 `#fff2f9` / 粉紫蓝 `#e1e1fe`
- 主强调：樱粉 `#e96ba8`
- 次强调：粉紫蓝 `#8b8bef`
- 三档语义色（我们加的，规范没写）：ok `#4fb38a` / warn `#f0a500` / err `#d04848`
- 自带 light/dark 双主题，顶栏右侧切换按钮

### 3.2 −12° 斜切（樱羽的记忆点）

用在：
- 顶栏品牌 mark 的方块
- 登录页左上角的品牌 mark
- KPI 卡片右上角的斜纹装饰
- 「最近异常」「系统健康」「调用量」卡片右上角的斜纹
- 进度条（Agent 占比）尾部斜切

整屏 ≤ 3 处的规则**严格执行**。

### 3.3 圆角只用三档

- `--radius-s 7px`（tag, input 内的小块）
- `--radius-m 10px`（按钮 / 输入框 / chip）
- `--radius-l 14px`（卡片）

**没有任何其它圆角值**。

### 3.4 强调色面积 ≤ 15%

- 主导航不铺色，只给 active 项一个浅粉底
- 卡片背景全白，强调色只出现在：按钮、chip、状态点、品牌 mark、装饰斜纹
- 用 1px 边框 (`var(--border)`) 分层，不靠重阴影

## 4. 响应式断点

```
≤ 760px        手机：顶栏折叠成汉堡菜单，KPI 2 列，技能行 3 行布局
761–1100px     平板：KPI 2 列，agents 单列，技能行 4 列
≥ 1101px       桌面：KPI 4 列，agents 2 列，技能行 6 列
```

技能行在手机上的多行布局：
```
┌───────────────────────────────────────┐
│ ⋮⋮  Device Status                  ⬤  │  ← 拖拽手柄 + 名字 + 启停开关
│       device_status                   │
│     查看所有上报设备的最新状态…       │  ← 描述
│     [本地] [只读]   [● 全部可见 ▾]    │  ← 标签 + 可见性下拉
└───────────────────────────────────────┘
```

详细媒体查询在 `src/styles/app.css` 末尾 `@media (max-width: 760px)`。

## 5. 数据流

```
┌──────────┐  fetch+Bearer   ┌──────────────────┐
│ React UI │ ──────────────► │ mcp-gateway      │
│          │ ◄────────────── │  /api/console/*  │
└──────────┘     JSON         └──────────────────┘
       │
       ├── localStorage: token, username, theme, skill-groups (兜底)
       │
       └── useAsync hook + ApiError 类
            ├─ 401 → 自动清 token → 跳登录
            └─ 网络错误 → 红底 toast
```

技能可见性的特殊数据流：
- 技能页要展示「这个技能对哪些 agent 可见」
- 但后端是按 agent 存白名单 (`/agents/:id/visibility`)
- 所以技能页加载时**并发请求**所有 agent 的 visibility，在前端做反向 join
- 用户点击下拉勾选时，**只对发生变化的 agent**调用 `setVisibility`

见 `src/pages/Skills.tsx > visForSkill / updateVisForSkill`。

## 6. 拖拽

用 HTML5 原生 drag&drop，不依赖第三方库（避免 react-dnd / dnd-kit 这种 200KB+ 的庞然大物）。
helper 在 `src/lib/drag.ts`，只有 30 行。支持：
- 拖技能在组内换位置
- 拖技能跨组（落到组的空白处 = 追加到末尾；落到某行 = 插入到那行上面）
- 拖组本身换顺序（拖 group-head 落到另一个 group-head 上）

数据 mime type `application/x-asashiki`，避免和其他 drag source 冲突。

## 7. 未来想加但目前没加的功能

| 功能 | 为什么没加 | 加的话怎么做 |
|---|---|---|
| 工具组（Access Groups）= 把工具打包给新 agent | 用户暂未明确要求 | 在 Skills 页加一列「所属工具组」chip + 一个"组"管理弹窗。后端需要新数据结构 |
| 每个工具的限流（rate limit） | 后端未实现 | 在技能行右侧加个「每分钟最多 N 次」字段。需后端配合 |
| 工具详情抽屉（点行展开看 inputSchema + 最近 5 次调用） | 减少首屏复杂度 | 点行触发 side drawer，前端组件已经留好位置（`SkillRow` 可改为 expandable） |
| `allow_all_keys` 公共工具开关 | 当前默认就是"全部可见"模式，不需要这个开关 | 一旦白名单模式变成默认，需要这个开关 |
