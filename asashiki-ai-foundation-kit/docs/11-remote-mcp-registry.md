# 11 Remote MCP Registry

## 1. 这层是干什么的

Remote MCP Registry 是 `core-api` 里新增的一层“上游 MCP 连接管理”。

它解决的是这类问题：

- 我现在又多了一个 MCP 地址，要怎么接进项目？
- 我想先看看这个 MCP 到底暴露了哪些工具
- 我想先测试工具能不能跑，再决定要不要做专门页面或专门包装

这层的目标不是“万能配置器”，而是：

- 先登记
- 先连通
- 先可视化
- 先可测试

## 2. 为什么这一步现在最合适

你现在最大的痛点不是少一个页面，而是：

- MCP 越来越多
- 每个 MCP 的能力都不一样
- 终端配置和项目内部结构你都不容易直接理解

所以当前最合理的方向，不是立刻做“所有 MCP 都能在 UI 里完整配置”，而是先让控制台具备：

- 看得见
- 看得懂
- 测得到

这一步做稳以后，再逐步决定哪些 MCP 适合抽成真正的控制台配置项。

## 3. 当前采用的结构

参考了两类成熟思路：

### 官方 MCP SDK

项目直接使用：

- `Client`
- `StreamableHTTPClientTransport`

来连接上游远程 MCP。

### MCPHub 一类项目的模式

它们通常把系统拆成：

- 配置注册表
- 客户端管理
- 工具目录发现
- 工具调用适配

当前仓库采取的是这个思路的简化版：

- `REMOTE_MCP_SERVERS_JSON`
  - 注册表
- `remote-mcp.ts`
  - 客户端管理 + 工具目录发现 + 工具调用
- `admin-web`
  - 查看与测试界面

## 4. 当前配置模型

每个远程 MCP 目前只要求最少信息：

```json
{
  "id": "supabase",
  "name": "Supabase Remote MCP",
  "url": "https://mcp.supabase.com/mcp?project_ref=xxx&read_only=true",
  "description": "只读 Supabase MCP",
  "bearerTokenEnv": "SUPABASE_MCP_ACCESS_TOKEN"
}
```

设计原则：

- `name` 和 `url` 是必需
- `description` 用人话说明用途
- `bearerTokenEnv` 只在需要时才加
- 敏感 token 不直接写进 JSON

## 5. 控制台现在能做什么

在 `连接中心` 里，现在可以：

- 看每个上游 MCP 是否在线
- 看认证模式
- 看它暴露了多少工具
- 看每个工具是不是只读
- 给单个工具填 JSON 参数并测试

这一步非常重要，因为你以后再接新的 MCP，不需要先猜它“到底有没有接通”。

## 6. 现在还故意没做什么

当前故意没有做：

- 控制台里直接新增任意 MCP
- 控制台里管理所有复杂认证流程
- 自动把所有上游工具重新暴露成你自己的 `mcp-gateway` 工具
- 按 UI 表单细分每个 MCP 的专属配置字段

原因很简单：

- 现在项目还在收敛阶段
- 你自己的 MCP 种类还没有稳定下来
- 先做复杂配置器，反而更容易把系统做乱

## 7. 适合你的实际使用方式

现阶段最适合你的方式是：

1. 你告诉我新的 MCP 是什么
2. 我帮你把它登记进项目
3. 你在控制台里看状态和测试结果
4. 如果它真的会长期用，再决定是否做成专门页面或专门配置项

这比一上来追求“所有 MCP 都可视化自助配置”更稳。

## 8. 和 Supabase 的关系

Supabase 只是第一个代表性案例，不是特例架构本身。

这层做好以后：

- 以后可以接 Supabase MCP
- 也可以接别的远程 HTTP MCP
- 甚至可以接你自己写的上游 MCP

换句话说：

这一步不是“给 Supabase 开个后门”，而是给整个项目补上了“连接上游 MCP”的通用能力。
