# 10 Admin-first Execution Plan

## 1. 这轮到底要解决什么

这一轮不再追求“多接几个新系统”，而是先解决两个更核心的问题：

- 让你主要通过 `admin-web` 使用这个项目
- 让外部连接器 / MCP 的接入与测试方式变得稳定、可重复

换句话说，这一轮的目标不是“功能越多越好”，而是：

**把项目推进成你能看得懂、用得上、以后能继续长大的控制台。**

## 2. 这轮的总原则

### 原则 A：`admin-web` 是主入口

后续优先做：

- 看数据
- 改数据
- 看连接状态
- 测试 MCP / 连接器

尽量减少你必须直接碰终端、SQLite、源码的情况。

### 原则 B：`public-web` 先冻结

目前不继续投入公开页面，不做正式对外发布，不动主站。

保留现状即可：

- `public-web` 只做本地预览
- 后面如果要公开展示，再从 `admin-web` 管理好的数据里导出或暴露给静态页面

### 原则 C：连接器先由 Codex 接入

当前阶段不做“所有 MCP 都能在控制台里自由配置”的大系统。

原因：

- 每个 MCP / 外部连接都不一样
- 如果一开始就做成复杂配置器，UI 很快会失控
- 你当前更需要“能接进来、能测试、能看状态”

因此这轮采用：

- 你告诉我一个连接器是什么
- Codex 负责把它接进项目
- 控制台负责查看、测试、启用/禁用

### 原则 D：简单连接以后再做 UI 配置化

如果后面发现一类连接器真的很统一，比如：

- 名称
- URL
- 一句话说明

那再把它们做成控制台可新增的“简单连接器”。

但这不是这一轮的首要目标。

## 3. 这一轮完成后，你应该得到什么

### 3.1 数据不再只能靠终端改

优先把这些做成控制台可见、可改：

- Profile Summary
- Journal Draft / Entry
- 公开状态相关的核心文本

### 3.2 控制台里能看到连接器

每个连接器至少应该在 UI 中显示：

- 名称
- URL
- 一句话说明
- 当前状态
- 是否启用
- 是否允许暴露给 agent
- 最近一次测试结果

### 3.3 控制台里能测试 MCP

控制台中应有一个 MCP Test Center，至少支持：

- 查看当前工具列表
- 查看工具用途
- 执行基础 smoke
- 显示成功/失败结果

### 3.4 第一个真实外部数据源落地

本轮最合适的真实试点是：

- Supabase 时间日志只读接入

因为它比“手机健康数据实时同步”简单得多，也更容易验证价值。

## 4. Supabase 试点怎么理解

### 用户侧输入尽量简化

对于你来说，第一版只需要提供：

- `name`
- `url`
- 一句话说明

例如：

- `supabase-time-log`
- 一个只读 URL
- “只读查询 `time_events`，用于回答某个时间点我在做什么”

### 系统侧如何处理

虽然你只输入这些，但底层接入仍由 Codex 负责完成，包括：

- 接口接入
- 查询逻辑
- 控制台展示
- MCP 工具映射
- 测试与审计

也就是说：

**你用人话描述连接器，Codex 负责把它翻译成系统行为。**

## 5. 这一轮的执行拆分

### 阶段 1：`admin-web` 先变成真正的控制台

优先做：

- Profile 编辑页
- Journal 管理页增强
- Connector Center
- MCP Test Center

### 阶段 2：把“查看状态”升级成“可测试”

重点是让你能在控制台中判断：

- 某个连接是不是活着
- 某个 MCP 工具是不是能用
- 某次测试为什么失败

### 阶段 3：接入第一个真实连接器

首选：

- Supabase 时间日志只读试点

目标不是一次做成超级通用系统，而是先跑通一个真实例子。

### 阶段 4：根据试点再决定哪些功能值得配置化

到那时再判断：

- 哪些字段适合给你在 UI 里填
- 哪些仍然应该留给 Codex 改代码

## 6. 本轮明确不做什么

- 不继续扩展 `public-web`
- 不正式发布 Public Web
- 不动 `asashiki.com` 主站
- 不正式公网发布 Admin Dashboard
- 不一次性把所有连接器做成通用配置器
- 不先上复杂记忆库系统
- 不先做健康数据实时同步

## 7. MCP / 连接器的现实做法

### 现在怎么做

当前阶段：

- 连接器由 Codex 接入
- 控制台负责查看和测试

### 以后怎么演进

如果某类连接器越来越统一，再把这类连接器做成控制台可配置。

也就是说，未来会变成两层：

- 简单连接器：控制台可配置
- 复杂连接器：继续由 Codex 接入

## 8. 推荐的日常工作流

### 系统升级

当你要新增功能、接入新连接器、改后端逻辑时：

1. 本地开发
2. 本地验证
3. `git push`
4. VPS 上 `git pull`
5. VPS 上重建服务
6. 跑 health / MCP smoke

### 日常数据维护

当项目把控制台补起来后，像这些事尽量在 `admin-web` 做：

- 改 profile
- 改公开文本
- 看连接状态
- 测试工具

## 9. 本轮完成标准

如果这一轮完成，应该达到下面这个效果：

- 你已经不需要主要依赖终端来管理文本数据
- 你能在控制台里看懂当前有哪些连接和 MCP
- 你能自己看见测试结果，而不是只听终端输出
- 你有了第一个真实数据源试点，而不是全是演示数据

## 10. 本轮验证清单

- `admin-web` 本地启动正常
- 核心页面能看到并编辑关键文本数据
- Connector Center 能看到已接入连接
- MCP Test Center 能执行基础 smoke
- Supabase 时间日志试点能回答至少一个真实问题
- 本轮代码完成后执行 `git push`
